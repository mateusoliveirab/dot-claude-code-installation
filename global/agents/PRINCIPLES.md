# Agentic Organism — Design Principles

This document is the constitution of the platform engineering agent ecosystem.
Every agent must be designed against these principles. It is also the reference for
creating new agents — whether written by a human or generated by another agent.

---

## What is this ecosystem?

A set of autonomous agents that run on a schedule, observe the state of the world
(codebase, infrastructure, dependencies, CI), act within their authority, and surface
what requires human judgment — without requiring human presence to function.

The ecosystem is designed to operate unattended from 00h–06h Mon–Fri.
The human sees a morning briefing. Not a list of failures. Not noise.
A curated signal of what needs attention — and confidence that everything else is handled.

**The daily-briefing agent is the synthesis layer.** It reads the output of all other
agents and produces the morning report. Design every agent with the briefing in mind:
if something is worth escalating, escalate it in a way the briefing can consume.

---

## The Five Pillars

### 1. Memory

Stateless agents are noisy after the first run and useless after the second.
Every agent must persist state across runs.

**State file location:** `.claude/state/<agent-name>.json` at the repo root.

**Mandatory state schema:**
```json
{
  "last_run": "<ISO 8601 timestamp>",
  "ignore": [],
  "backlog": [],
  "escalation": []
}
```

- `last_run` — enables diff-driven scope. Always update at the end of a successful run.
- `ignore` — human-editable. The agent reads and respects it without question.
- `backlog` — agent-managed. Tracks findings the agent could not resolve autonomously.
- `escalation` — agent-managed. Items for the daily-briefing to surface.

The state file is committed to the repo at the end of each run so it persists
across environments and machines.

---

### 2. Autonomy Tiers

Every finding must map to exactly one tier. No gray zones.
This is the decision engine that enables unattended operation.

| Tier | Label | Agent action | Human required? |
|------|-------|-------------|-----------------|
| 1 | **Auto-fix** | Apply change, open PR | No |
| 2 | **Apply + flag** | Apply change, open PR with explicit warning | No — but informed |
| 3 | **Flag only** | Open issue, add to backlog | Yes — to decide |
| 4 | **Escalate** | Add to escalation array → daily-briefing | Yes — urgent |

**Rules for tier assignment:**
- If the change is reversible, safe, and tests pass → Tier 1 or 2
- If the change requires human judgment (breaking, ambiguous, risky) → Tier 3
- If a Tier 3 item has been unresolved for more than `escalation_threshold_days` → Tier 4
- If the finding is critical and time-sensitive → Tier 4 immediately (no waiting period)

**Every agent must declare its tier boundaries explicitly** in its own PRINCIPLES section.
A reader should be able to determine in 10 seconds what the agent will and will not do autonomously.

---

### 3. Zero Noise

An agent that generates noise gets disabled. This is the primary threat to ecosystem health.

**Zero noise rules:**
- Never open a duplicate issue or PR. Check backlog and open branches before acting.
- Never re-flag a finding that is in the `ignore` list.
- Never re-flag a finding already in `backlog` with `issue_opened: true` — update it instead.
- If nothing changed since last run, update `last_run` and exit silently.
- Prefer updating an existing issue (new comment or updated body) over opening a new one.

**The ignore list is the human veto.** When a human closes an issue and adds it to
`ignore`, the agent must never reopen it. The ignore list is the contract between
the agent and the human.

Ignore entry format:
```json
{
  "type": "package | resource | file | subproject | cve | custom",
  "value": "<identifier>",
  "reason": "<why this is being ignored>",
  "until": "<ISO date — optional, omit for indefinite>"
}
```

At the start of each run, the agent must expire entries whose `until` date has passed
and log: `"Ignore entry for <value> expired on <date> — re-activating"`

---

### 4. Escalation Chain

Agents do not operate in isolation. They communicate through the state file's
`escalation` array, which the `daily-briefing` agent reads every morning.

**Escalation entry format:**
```json
{
  "severity": "critical | high | medium",
  "type": "<agent-specific category>",
  "subject": "<what is affected>",
  "summary": "<one sentence — what is wrong and why it matters>",
  "days_open": 0,
  "reference": "<issue number, PR, or file>",
  "first_escalated": "<ISO timestamp>"
}
```

**Escalation triggers:**
- Any Tier 4 finding → escalate immediately
- Any Tier 3 item in backlog older than `escalation_threshold_days` (default: 7) → escalate
- Any finding with a critical severity tag → escalate regardless of age

**The daily-briefing agent must:** read the `escalation` array of every agent's
state file, deduplicate by `subject`, sort by `severity`, and include in the morning report.

---

### 5. Actionable Outputs

Every issue, PR, or escalation must be immediately actionable by a human with no
additional research required.

**PR requirements:**
- What changed and why (not just a diff)
- Test results (passed / failed / no test suite)
- License or breaking change flags if applicable

**Issue requirements:**
- Exact evidence (plan output, audit JSON, test failure log)
- Root cause explanation
- Step-by-step resolution instructions
- Migration guide if breaking change (fetched from changelog, not invented)
- Links to relevant docs

**Never write:** "you should consider updating X" or "this may require attention".
Write: "run `terraform import aws_s3_bucket.logs logs-bucket-prod` to reconcile" or
"update line 42 of `apps/api/src/db.ts` to use the new `createClient()` API".

## Mandatory Agent Structure

Every agent file must contain:

```
---
name: kebab-case-name
description: <What it does> + <When it runs>
---

## Principles         ← domain-specific values
## Boundaries         ← explicit list of what it must NOT do
## Autonomy Tiers     ← what it acts on vs flags vs escalates
## State File         ← schema for its .claude/state/<name>.json
## Workflow           ← numbered steps, in order
## Completion         ← structured summary output
```

The `Boundaries` section is non-negotiable. It must list at minimum:
- What files/systems it will never modify
- Whether it commits to main (almost always: never)
- Whether it takes destructive actions (almost always: never)

---

## Diff-Driven Scope

By default, agents process only what changed since `last_run`.

```bash
git log --since="<last_run>" --name-only --pretty=format: | sort -u
```

**Exception:** security and compliance checks (CVE audits, drift detection, secret scanning)
must always run full scope regardless of last_run. New vulnerabilities appear in
unchanged code.

**First run:** always full scope. Set `last_run = null` in a missing state file.

---

## Idempotency Contract

Running any agent twice in a row must produce the same result.
The second run must not create additional PRs, issues, or commits beyond
updating `last_run` in the state file.

Checklist:
- [ ] Check open branches before creating a new one
- [ ] Check open issues before creating a new one
- [ ] Check backlog before adding a new entry
- [ ] Deduplication logic is explicit, not assumed

## Creating a New Agent

When designing a new agent, answer these questions first:

1. **What is its single responsibility?** One agent, one domain.
2. **What are its tier boundaries?** What will it fix silently? What will it only flag?
3. **What is its blast radius?** What is the worst thing it could do if it misbehaves?
4. **What does its state file look like?** What does it need to remember across runs?
5. **What does it escalate?** What should the human see in the morning briefing?
6. **What are its hard boundaries?** List the things it must never do.
7. **How does it integrate with daily-briefing?** What goes into the escalation array?

### Research phase — before writing a single line

Before designing the workflow, research what the community is already doing in this domain:

- Search for open-source tools, GitHub Actions, CLI utilities, and agent frameworks that solve similar problems
- Look at how teams at companies like GitHub, Stripe, Shopify, and Netflix approach the same domain (engineering blogs, conference talks, OSS repos)
- Check CNCF, OWASP, and similar foundations for established standards and tooling
- Search for "awesome-<domain>" lists on GitHub

**How to evaluate what you find — Quality Score:**

Score each practice or tool on 5 dimensions (1–3 each, max 15):

| Dimension | 1 | 2 | 3 |
|---|---|---|---|
| **Adoption** | Personal project / unknown | Mid-size team or company | Industry standard (CNCF, OWASP, Fortune 500) |
| **Maintenance** | Abandoned / last commit >2y | Active but slow | Actively maintained, recent releases |
| **Fit** | Tangentially related | Solves part of the problem | Directly addresses the agent's domain |
| **Autonomy-friendly** | Requires interactive input | Partially automatable | Fully scriptable, JSON output, CI-native |
| **Signal quality** | High false positive rate | Moderate noise | Low noise, high precision |

**Threshold:** Score ≥ 10 → include as primary tool. Score 7–9 → include as fallback. Score < 7 → skip.

This is how we decided on Gitleaks over grep, Semgrep over manual pattern matching, and Checkov over ad-hoc terraform linting — each scored 12–14 vs 5–7 for the alternatives.

A new agent is production-ready when:
- It has a state file with `last_run`, `ignore`, `backlog`, `escalation`
- Its autonomy tiers are explicitly documented
- Its boundaries section lists at least 4 hard constraints
- It is idempotent (running twice produces no side effects)
- It updates `last_run` at the end of every successful run
- It respects the `ignore` list without question
- It produces a structured summary at completion

## Anti-Patterns

These patterns kill autonomous agents. Avoid them.

| Anti-pattern | Why it kills the agent |
|---|---|
| No state file | Reruns everything, generates duplicate noise |
| No ignore list | Human cannot veto findings — agent gets disabled |
| Opening duplicate issues | Noise. Destroys trust. |
| Vague issue bodies | Human has to do more research than the agent — why use it? |
| Fixing things outside tier authority | Unpredictable blast radius — agent gets disabled |
| Running tests with side effects | Breaks prod data, sends emails, charges APIs |
| Using `--force` flags | Irreversible damage |
| Blocking entire run on one failure | One broken module kills the whole audit |
| Reporting without context | "Drift detected" is useless. "sg-0abc was modified externally — SSH opened to 0.0.0.0/0" is actionable |
| Silent success when there is nothing to do | Always update `last_run`. Always output a completion summary. |

---

## Ecosystem Health Invariants

These must always be true:

1. The daily-briefing on any given morning must reflect the actual state of the system — no stale data
2. Any critical finding (CVE 9+, prod drift, exposed secret) reaches a human within 24h regardless of what other agents do
3. An agent that fails mid-run must not leave the repo in a dirty state (uncommitted changes, orphaned branches)
4. No agent ever silently succeeds on its second run after a failure — it must log what it skipped and why
5. The `ignore` list is the only mechanism by which findings are permanently suppressed — agents must not implement their own suppression logic outside of it
